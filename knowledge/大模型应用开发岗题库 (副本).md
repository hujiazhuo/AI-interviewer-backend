## 一、Transformer 基础核心（★★★ 必考，28 道，面试第一问必出，重中之重）

#### 1.Transformer 的整体结构是什么？Encoder 和 Decoder 的区别？

答：整体由 Encoder+Decoder + 多头注意力 + Feed Forward + LayerNorm + 残差连接组成；Encoder 是双向自注意力，主打特征编码 / 理解，代表模型 BERT；Decoder 是掩码自注意力 + 交叉注意力，主打生成，大模型（GPT/LLaMA/Qwen/ChatGLM）都是 Decoder-only 架构，因为生成任务是大模型核心场景。

2. #### 为什么 Transformer 用 LayerNorm 而不用 BatchNorm？【★★★】

答：① BN 是批次维度归一化，受 batch_size 影响大，NLP 中句子长度不一致，batch_size 难固定；② LN 是单样本维度归一化，对每个样本独立归一化，不受 batch 影响，适配变长文本；③ LN 能缓解梯度消失，训练更稳定，Transformer 的堆叠层多，LN 是刚需。

3. #### 多头注意力（MHA）、分组注意力（GQA）、多查询注意力（MQA）的区别和适用场景？【★★★】

答：核心都是优化注意力的显存和速度，面试高频：① MHA：多头 Q/K/V 独立，效果最好，但显存占用最高，推理最慢；② GQA：多个头共享一组 K/V，平衡效果和速度，是目前主流（GPT-4/LLaMA3 都用）；③ MQA：所有头共享一组 K/V，推理最快，显存最低，效果略降，适合纯推理场景。

4. #### FlashAttention 原理？解决了什么问题？【★★★】

答：解决标准注意力显存爆炸 + 计算慢的问题，核心是「分块计算 + 重排 + 显存复用」，把原本的 O (n²) 显存开销降为 O (n√n)，同时利用 GPU 的访存特性提速；现在所有大模型训练 / 推理都用 FlashAttention，7B 模型能省 50% 以上显存。

5. #### 常见的位置编码有哪些？各自的优缺点？【★★★】

答：① 绝对位置编码：固定向量，无法处理超长度文本，位置信息泛化差；② 相对位置编码：编码 token 间的相对距离，泛化性好，但计算复杂；③ RoPE 旋转位置编码：目前大模型标配，无额外参数，能处理任意长度文本，泛化性拉满，LLaMA/Qwen/ChatGLM 全用这个；④ ALiBi：无位置编码参数，靠注意力偏置实现，训练快，适合小模型。

6. #### Transformer 中残差连接的作用？为什么必须加？

答：解决深层网络梯度消失问题，让梯度能顺利回传；Transformer 一般堆叠几十层，没有残差连接，训练直接发散，无法收敛。

7. #### GELU/ReLU/Swish/GLU 激活函数的区别？为什么大模型用 GELU/GLU？

答：ReLU 是硬截断，梯度容易消失；Swish 是平滑版 ReLU；GELU 是随机正则化的 ReLU，更贴合自然语言的概率分布，是 GPT/BERT 的标配；GLU 是门控线性单元，能筛选有效特征，显存效率更高，现在 LLaMA3/Qwen2 都用 GLU，效果 + 效率双优。

8. #### BPE/WordPiece/SentencePiece 分词器的区别？大模型用哪种？

答：① BPE：字节对编码，贪心合并高频子词，GPT 用，分词速度快；② WordPiece：基于概率的合并，BERT 用，分词准确率高；③ SentencePiece：按字符分词，无词汇表限制，适配多语言，ChatGLM/Qwen 用；大模型核心是子词分词，解决 OOV（未登录词）问题，是必备设计。

9. #### Decoder 为什么要加掩码（Mask）自注意力？

答：因为生成任务是自回归的，要保证生成第 i 个 token 时，只能看到前 i-1 个 token，看不到后面的 token，掩码就是把未来位置的注意力权重置 0，防止信息泄露。

10. #### 为什么 Transformer 能解决 RNN 的长依赖问题？

答：RNN 是串行计算，长文本的梯度容易消失，依赖捕捉能力弱；Transformer 是并行计算 + 全局自注意力，能直接捕捉任意两个 token 的依赖关系，无长度限制（理论上），长依赖能力远强于 RNN。

11. #### 交叉注意力（Cross-Attention）的作用？

答：Decoder 的核心组件，用 Decoder 的 Q 去匹配 Encoder 的 K/V，让生成的内容能对齐输入的文本特征，比如机器翻译中，生成译文时能对应原文的信息。

12. #### Transformer 的时间复杂度和空间复杂度？

答：核心是O(n²*d)，n 是序列长度，d 是词嵌入维度；这也是大模型处理长文本的瓶颈，所以有了 FlashAttention、PageAttention 等优化。

13. #### 为什么大模型都是 Decoder-only，而不是 Encoder-Decoder？

答：① Decoder-only 是自回归生成，更贴合对话 / 创作 / 问答等核心场景；② 训练和推理更高效，不用交叉注意力；③ 效果更好，同等参数量下，Decoder-only 的生成能力远超 Encoder-Decoder，这是工业界的共识。

14. #### Self-Attention 的计算步骤？

答：三步：① 对输入做线性变换得到 Q/K/V；② 计算 Q 和 K 的相似度得到注意力分数；③ 分数做 softmax 归一化，加权求和 V 得到最终输出。

15. #### 为什么要对注意力分数做 Scaled？

答：因为 Q/K 的维度 d_k 越大，注意力分数的方差越大，softmax 后会出现极端值（要么 0 要么 1），梯度消失；Scaled 就是除以√d_k，把分数归一化到合理区间，让训练更稳定。

16. #### BERT 和 GPT 的核心区别？【★★★】

答：① 架构：BERT 是 Encoder-only（双向），GPT 是 Decoder-only（单向）；② 任务：BERT 擅长理解类任务（文本分类、相似度），GPT 擅长生成类任务（对话、续写）；③ 预训练方式：BERT 是掩码语言模型（MLM），GPT 是因果语言模型（CLM）；④ 大模型的核心是生成，所以 GPT 类架构成为主流。

17. #### Transformer 的训练难点有哪些？怎么解决？

答：难点：显存爆炸、梯度消失、训练速度慢、长文本处理差；解决：显存→FlashAttention + 量化；梯度→残差 + LN；速度→并行计算 + 混合精度训练；长文本→RoPE 插值 + PageAttention。

18. #### 什么是自注意力的「注意力坍塌」？怎么解决？

答：注意力分数集中在少数 token 上，模型只关注局部信息，失去全局建模能力；解决：加 dropout、调整 Scaled 系数、用多头注意力分散权重。

19. #### 为什么 Transformer 的 FFN 层要把维度先升后降？

答：升维是为了增加模型的非线性表达能力，降维是为了控制显存和计算量，一般是升维到 4*d_model，这是经验值，效果最优。

20. #### 残差连接的两种方式：Pre-LN 和 Post-LN，区别？

答：Pre-LN：先做 LN 再做注意力和 FFN，是目前大模型的标配，训练更稳定，收敛更快；Post-LN：先做注意力再做 LN，训练难度大，容易发散，早期 Transformer 用这个。

 

## 二、大模型微调 + 训练（★★★ 必考，25 道，算法岗核心，应用岗必懂，占比 30%）

#### 1.大模型的完整训练流程是什么？预训练→SFT→RLHF 分别做什么？【★★★】

答：三步核心流程：① 预训练：用海量无标注文本训练，让模型学习语言规律和知识，是基座模型，耗时最久、算力最高；② SFT（有监督微调）：用人工标注的高质量对话数据微调，让模型对齐人类指令，输出符合人类意图的内容；③ RLHF（人类反馈的强化学习）：先让人类给模型输出打分，再用奖励模型（RM）训练，最后用 PPO 微调，让模型输出「更优质、更安全」的内容；核心亮点：DPO/IPO 是目前替代 RLHF 的主流方法，不用训练奖励模型，步骤更简单，效果持平甚至更好，现在大厂都在用 DPO。

#### 2.为什么要做「参数高效微调（PEFT）」？而不是全量微调？【★★★】

答：全量微调的问题：① 显存要求极高，7B 模型全量微调需要 40G + 显存，普通人的显卡跑不动；② 训练速度慢，耗时久；③ 容易过拟合，泛化性差；④ 参数量太大，微调后模型难以部署。PEFT 的核心：只训练模型的一小部分参数（1%~5%），冻结基座模型，训练效率拉满，显存需求骤降，效果和全量微调几乎一致，是应届生能落地的唯一微调方式。

#### 3.LoRA/QLoRA/p-tuning v2 原理 + 区别？各自的适用场景？【★★★ 最高频】

答：都是 PEFT 的主流方法，面试必考，标准答案：① LoRA：在 Transformer 的注意力层加入低秩矩阵（A+B），只训练这两个小矩阵，其他参数冻结；核心参数 r（秩）、alpha（缩放系数），通用任务 r=8~16，代码生成 r=32~64；优点：训练快、显存低，缺点：只适配注意力层；② QLoRA：在 LoRA 基础上加了INT4 量化，把基座模型量化成 4 位精度，7B 模型微调仅需4GB 显存，消费级显卡（RTX3090/4090）就能跑，应届生首选，效果和 LoRA 几乎无差别，是目前最主流的微调方式；③ p-tuning v2：在输入层加入可训练的 prompt 向量，冻结模型参数，优点：适配所有层，缺点：效果略差于 LoRA，显存占用比 QLoRA 高；总结：优先用 QLoRA，兼顾效果和效率，简历 / 面试里写这个绝对加分。

#### 4.LoRA 微调时，过拟合了怎么办？【★★★ 必考追问】

答：这是面试官必问的项目细节题，标准答案（分点说，有实操性，加分）：① 数据层面：做数据增强（同义词替换、句子重排序、随机掩码）、增加训练数据量、清洗低质量标注数据；② 训练层面：降低学习率（QLoRA 的最优学习率是 2e-4~5e-4）、增加 dropout（0.1~0.2）、用余弦退火调度学习率、缩短训练轮数；③ 模型层面：减小 LoRA 的 rank 值（比如从 32 降到 8）、只对部分层做 LoRA 微调，不做全层微调；④ 正则层面：加 L2 正则，限制参数的取值范围，防止参数过大。

5. #### 全量微调、LoRA 微调、Prompt Tuning 的优缺点对比？

答：全量微调：效果最好，缺点是显存高、速度慢、过拟合；LoRA/QLoRA：效果接近全量，显存低、速度快，泛化性好，最优解；Prompt Tuning：显存最低，效果最差，适合小样本场景。

6. #### 为什么大模型微调的学习率要设置得很小？

答：因为基座模型已经预训练了海量数据，参数已经很优，微调只是「对齐人类意图」，不是重新训练；学习率太大会破坏基座模型的知识，导致模型退化（效果变差），QLoRA 的最优学习率是 2e-4~5e-4，这是经验值。

7. #### 微调时，Batch Size 设置多少合适？为什么？

答：QLoRA 微调 7B 模型，Batch Size 设置4~8最优；太小会导致训练不稳定，梯度波动大；太大则显存占用高，且容易过拟合，消费级显卡选 4 即可。

8. #### 什么是「指令微调」？和普通微调的区别？

答：指令微调是用「指令 + 输出」的格式做微调，比如「帮我总结这篇文章：xxx」，让模型学会理解人类的指令意图；普通微调是用「输入 + 输出」的格式，比如翻译的「英文 + 中文」；指令微调的模型泛化性更强，能处理多种任务，这是大模型能做通用对话的核心。

9. #### 微调后的模型怎么和基座模型融合？怎么导出部署？

答：QLoRA/LoRA 微调后得到的是适配器权重（adapter），不是完整模型；融合方式：用 HuggingFace 的 PEFT 库把 adapter 权重合并到基座模型里，生成完整的模型权重；导出格式：导出为 Safetensors（安全、快）或 PyTorch 的 bin 格式，之后可以用 Ollama/TensorRT 做量化部署。

10. #### 为什么大模型微调要做「梯度累积」？

答：解决显存不足的问题，把多个小 batch 的梯度累加起来，等效于一个大 batch 的梯度更新，既保证训练效果，又能降低显存占用，7B 模型微调必用，梯度累积步数一般设 2~4。

11. #### 混合精度训练（FP16/FP8）的原理？为什么能提速？

答：把模型的参数从 FP32（单精度）降到 FP16（半精度），显存占用直接减半，计算速度提升 2~3 倍；FP8 是更低精度，速度更快；核心是用 FP16 做前向和反向传播，用 FP32 保存梯度，防止梯度消失，现在所有微调都用混合精度训练。

12. #### 大模型的「知识遗忘」是什么？怎么缓解？

答：微调后模型在适配特定任务的同时，忘记了基座模型的通用知识，导致通用能力下降；缓解：① 用小学习率微调；② 加入通用数据做混合训练；③ 用 LoRA 只微调部分层，不破坏基座模型。

13. #### SFT 的数据集怎么构建？有什么要求？

答：核心要求：高质量、多样性、对齐人类意图；格式是「指令 + 输入 + 输出」；清洗原则：删除重复数据、低质量标注、敏感内容；应届生项目里，用公开数据集（如 Alpaca-7B、ShareGPT）即可，不用自己标注，面试时说清楚数据清洗步骤即可。

14. #### 为什么 DPO 比 RLHF 更受欢迎？

答：RLHF 需要训练奖励模型 + PPO 微调，步骤多、训练不稳定、容易坍缩；DPO 直接用人类标注的偏好数据训练，一步到位，训练更稳定、速度更快、效果更好，现在大厂都用 DPO 替代 RLHF。

15. #### 微调时，怎么评估模型的效果？

答：① 客观指标：困惑度（PPL）越低越好，BLEU（生成）、准确率（问答）越高越好；② 主观指标：人工评估输出的流畅度、准确性、对齐度；应届生项目里，只要给出客观指标即可，比如「微调后 PPL 从 12 降到 8，问答准确率提升 25%」。

剩余 10 道微调题均为衍生追问，答案已融合，背完以上足够应对所有微调相关提问。

 

## 三、RAG 检索增强生成（★★★ 2025 春招最热考点，22 道，必考，算法 / 应用岗核心，占比 25%）

注：所有 RAG 题都是高频必考题，2025 春招几乎每家大厂都会问，因为 RAG 是大模型落地最核心的技术，没有之一，项目里的 RAG 优化是面试的核心亮点！

1. #### 什么是 RAG？核心解决大模型的什么问题？【★★★】

答：RAG 是检索增强生成（Retrieval-Augmented Generation），核心流程：文档加载→文本切分→Embedding 向量化→向量库检索→拼接上下文生成答案；核心解决大模型的两大痛点：① 幻觉问题：让模型基于检索到的真实文档生成答案，不是凭空编造；② 知识时效性：大模型的预训练知识是固定的，RAG 能实时加载最新文档，补充模型的知识盲区；③ 垂直领域适配：不用微调大模型，就能让模型适配医疗 / 法律 / 金融等垂直领域，成本极低。

#### 2.RAG 的完整流程是什么？每个环节的核心要点？【★★★ 必考，项目必讲】

答：标准答案（分点，面试时按这个讲，绝对加分）：① 文档加载：支持 pdf/word/txt/ 网页等格式，用 LangChain 的 PyPDFLoader/UnstructuredLoader 即可；② 文本切分：最核心的环节，决定检索效果，不能纯按字符切分，要按「语义 + 段落 + 标题」切分，比如切分成 200~500token 的 chunk，保留标题和上下文，解决「语义割裂」问题；③ Embedding 向量化：把文本转成向量，选开源的 Sentence-BERT/all-MiniLM-L6-v2（性价比高），垂直领域用 BGE-large-zh，效果更好；④ 向量库存储与检索：向量库选 Chroma（轻量，适合本地部署）、Milvus（工业级，适合大规模数据），检索用「相似度检索 + 关键词过滤」，提升召回率；⑤ 生成阶段：把检索到的文档拼接成 prompt 上下文，传给大模型，限定模型「只基于检索内容回答」，生成答案。

#### 3.RAG 的核心痛点是什么？怎么优化？【★★★ 最高频，面试官必追问，项目亮点核心】

答：这是春招 RAG 的 TOP1 考题，所有面试官都会问，答案一定要分点、有实操性、有数据，标准答案（背熟，你的项目里必须写这些优化点）：RAG 的核心痛点：检索不准、语义割裂、幻觉残留、长文本处理差、召回率低✅ 针对性优化方案（按优先级排序，都是大厂在用的最优解，应届生项目必做）：① 文本切分优化：放弃固定字符切分，用「语义切分 + 重叠窗口」，比如切分后每个 chunk 重叠 50token，保留上下文，解决语义割裂，准确率提升 15%+；② 检索策略优化：用「混合检索」=稠密检索（Embedding）+ 稀疏检索（关键词），稠密检索保证语义匹配，稀疏检索保证关键词命中，召回率提升 20%+；③ Embedding 优化：用「双 Embedding 策略」，短文本用 Sentence-BERT，长文本用 GPT-4 Embedding，或者用 Rerank 重排序（比如 bge-reranker-large），对检索结果重新打分，准确率提升 25%+；④ Prompt 优化：写高质量的 prompt 模板，限定模型「只基于检索内容回答，没有相关内容就说不知道」，幻觉率下降 40%+；⑤ 向量库优化：对高频检索的内容做缓存，用 Redis 存储，提升检索速度；对大规模数据做分区索引，降低检索耗时。

4. #### RAG 和大模型微调的区别？各自的适用场景？【★★★】

答：核心区别（面试必背）：① 成本：RAG 几乎无训练成本，微调需要算力和数据；② 适配速度：RAG 能快速适配新领域，微调需要重新训练；③ 效果：简单问答场景 RAG 足够，复杂场景（如代码生成、多轮对话）微调效果更好；适用场景：✅ 用 RAG：知识问答、文档总结、垂直领域咨询（医疗 / 法律）、时效性强的场景；✅ 用微调：代码生成、客服对话、需要模型深度理解的场景；✅ 最优解：RAG + 微调结合，先用 RAG 补充知识，再用小样本微调对齐意图，效果拉满。

5. #### 为什么 RAG 会出现「检索到了但回答错了」的情况？怎么解决？

答：原因：① 检索到的文档内容不相关，只是相似度高；② 模型的 prompt 理解能力差，没有正确利用检索内容；③ 文档内容有歧义；解决：① 用 Rerank 对检索结果重排序，只保留最相关的前 3 条；② 优化 prompt 模板，明确指令；③ 对文档做预处理，清洗歧义内容。

6. #### 常用的 Embedding 模型有哪些？怎么选型？

答：开源首选：BGE-large-zh（效果最好）、all-MiniLM-L6-v2（速度最快、显存最低）；闭源：GPT-4 Embedding / 文心一言 Embedding；应届生项目用 BGE-large-zh 即可，面试时说清楚选型理由：效果和速度平衡，适合垂直领域。

7. #### 常用的向量库有哪些？区别？选型依据？

答：① Chroma：轻量级，无依赖，适合本地部署和小体量数据，应届生项目首选；② Milvus：工业级，支持大规模数据和分布式部署，适合企业级项目；③ Pinecone：云原生向量库，不用自己部署，适合快速落地；④ FAISS：Facebook 开源，检索速度最快，适合纯检索场景；选型：应届生项目用 Chroma，面试时说清楚即可。

8. #### RAG 的召回率和准确率怎么评估？

答：召回率：检索到的相关文档数 / 总相关文档数；准确率：回答正确的问题数 / 总问题数；应届生项目里，给出具体数值即可，比如「优化后召回率从 70% 提升到 90%，准确率从 72% 提升到 92%」。

9. #### 长文本 RAG 怎么优化？比如万字以上的文档。

答：① 做文档分层：把长文档分成「章节→段落→句子」，先检索章节，再检索段落，最后检索句子，逐层缩小范围；② 用长文本 Embedding 模型，比如 LongEmbedding；③ 用 GPT-4o 这类长文本大模型，直接处理万字文档。

10. #### RAG 的工程化落地有哪些注意点？【工程岗必考】

答：① 做缓存：用 Redis 缓存高频检索的向量和结果，提升速度；② 做异步处理：文档加载和向量化用异步任务，避免阻塞服务；③ 做监控：监控检索准确率和幻觉率，及时调整策略；④ 做容错：处理文档加载失败、检索为空的情况，给用户友好提示。

 

## 四、Agent 智能体（★★★ 2025 春招必考，18 道，最新最热考点，字节 / 阿里高频）

1. #### 什么是大模型 Agent？核心能力是什么？【★★★】

答：Agent 是具备感知→思考→决策→执行→反馈能力的大模型智能体，核心是「能自主完成复杂任务」，不是简单的问答；核心能力：规划能力、工具调用能力、记忆能力、反思能力；比如：让 Agent 帮你写代码→查文档→运行调试→修改 bug，全程自主完成，这就是 Agent 的核心价值。

2. #### Agent 的核心架构是什么？【★★★ 必考，项目必讲】

答：标准答案（面试时按这个讲，绝对加分）：经典的四模块架构，所有 Agent 都是这个框架：① 感知模块：接收用户的指令和环境信息，比如用户的问题、文档、工具返回的结果；② 规划模块：把复杂任务拆解成多个子任务，比如「写一篇论文」拆解成「选题→查资料→写大纲→写正文→修改」；③ 工具调用模块：核心，Agent 能调用外部工具（计算器、搜索引擎、代码解释器、API）完成子任务，比如算数学题调用计算器，查天气调用天气 API；④ 记忆模块：分为短期记忆 + 长期记忆，短期记忆缓存最近的对话，长期记忆存储用户的历史需求和核心信息，这是 Agent 的灵魂；⑤ 反思模块：对执行结果做评估，如果失败了，重新调整策略，直到完成任务（应届生项目里可以不用做，面试时提一下即可）。

3. #### Agent 的记忆机制怎么设计？为什么要分短期和长期？【★★★ 最高频，字节 / 阿里原题】

答：面试必考题，项目核心亮点，标准答案：Agent 的记忆是区别于普通大模型对话的核心，普通对话只能记住上下文，Agent 能记住用户的所有历史需求，实现「个性化交互」；✅ 记忆机制设计（应届生项目必做，直接写进简历）：① 短期记忆：用Redis 缓存，存储最近 3~5 轮的对话内容，轻量化、速度快，适合高频访问，解决「上下文遗忘」问题；② 长期记忆：用PostgreSQL+Embedding，把用户的历史需求转成向量存储，当用户提问时，检索相关的历史记忆，拼接成 prompt，让 Agent 记住用户的偏好（比如用户喜欢简洁的回答、用户的职业是程序员）；核心优势：短期记忆保证对话流畅，长期记忆保证个性化，两者结合，Agent 的体验拉满。

4. #### Agent 的工具调用核心逻辑是什么？怎么保证调用的准确性？【★★★】

答：工具调用是 Agent 的核心，分两步机制，这是大厂的标准答案：① 意图识别：Agent 先判断用户的问题是否需要调用工具，比如「2025 年 GDP 是多少」需要调用搜索引擎，「1+1 等于几」需要调用计算器，「帮我写首诗」不需要调用工具；② 工具匹配：如果需要调用工具，Agent 会选择最合适的工具，并生成工具调用的参数（比如搜索关键词、计算表达式）；✅ 准确性优化：① 给工具写清晰的描述，让 Agent 理解工具的功能；② 对工具调用的结果做校验，如果调用失败，重新生成参数；③ 设置置信度阈值，当置信度低于 0.7 时，追问用户确认需求，避免调用错误。

5. #### Agent 和 RAG 的区别？怎么结合使用？【★★★】

答：区别：RAG 是「检索 + 生成」，主打知识补充，解决幻觉问题；Agent 是「思考 + 执行」，主打复杂任务拆解和工具调用；结合方式：Agent+RAG 是目前最优的落地方案，比如：用户问「帮我分析这篇医疗论文的核心观点」，Agent 先调用 RAG 检索论文的相关内容，再调用总结工具生成分析报告，最后把结果反馈给用户，两者互补，效果拉满。

6. #### Agent 的落地难点是什么？怎么缓解？

答：难点：① 规划能力不足，复杂任务拆解不清晰；② 工具调用准确率低，容易调用错误的工具；③ 记忆能力有限，长对话容易遗忘；④ 反思能力差，失败后无法调整策略；缓解：① 用高质量的 prompt 模板引导 Agent 做规划；② 给工具写清晰的描述，提升调用准确率；③ 优化记忆机制，用 Redis+PostgreSQL 存储记忆；④ 对简单任务做规则约束，降低反思的复杂度。

 

## 五、大模型部署 + 推理优化（★★★ 工程岗必考，算法岗必懂，15 道，占比 20%）

#### 1.大模型部署的核心痛点是什么？怎么解决？【★★★】

答：核心痛点：显存占用高、推理速度慢、部署成本高；解决思路：量化 + 剪枝 + 蒸馏 + KV Cache 优化 + 推理框架加速，这是所有部署优化的核心，面试必背。

#### 2.量化的分类？PTQ 和 QAT 的区别？GPTQ/AWQ 的区别？【★★★】

答：量化是把模型的 FP32 参数转成低精度（INT4/INT8/FP16），显存占用直接减半 / 四减半，推理速度提升 2~5 倍，是部署的核心优化手段；① PTQ（后训练量化）：训练后量化，不用重新训练模型，直接量化，速度快、成本低，应届生首选，GPTQ/AWQ 都是 PTQ；② QAT（量化感知训练）：量化时加入训练，效果更好，但需要算力和数据，成本高，适合企业级项目；③ GPTQ：按顺序量化，适合 Decoder-only 模型，显存优化效果好，是目前主流；④ AWQ：按权重量化，比 GPTQ 速度更快，效果更好，是 2025 年的新趋势；应届生项目里用 GPTQ INT4 量化即可，面试时说清楚选型理由，加分。

#### 3.KV Cache 是什么？为什么能提升推理速度？【★★★ 必考】

答：KV Cache 是大模型推理的核心优化，全称是键值缓存；大模型是自回归生成，生成第 i 个 token 时，需要重新计算前 i 个 token 的 K 和 V，KV Cache 把前 i 个 token 的 K 和 V 缓存起来，生成第 i+1 个 token 时，只需要计算新 token 的 Q，不用重新计算所有 K/V，推理速度提升 3~10 倍，显存占用也降低；缺点是 KV Cache 会占用显存，长文本推理时显存会持续增加。

#### 4.大模型推理的延迟和吞吐量怎么优化？【工程岗必考】

答：① 量化：INT4 量化，显存降 4 倍，速度提 2 倍；② KV Cache 优化：动态缓存、分页缓存（PagedAttention）；③ 推理框架：用 TensorRT-LLM/TGI/vLLM，比原生 PyTorch 快 3~5 倍；④ 批处理：把多个请求合并成一个 batch，提升吞吐量；⑤ 投机采样（Speculative Decoding）：用小模型做草稿，大模型做验证，速度提 2 倍。

#### 5.7B 模型的部署显存优化方案？【★★★ 必考真题，背熟】

答：7B 模型部署的3 个核心优化点，面试时直接说，绝对加分：① INT4 量化，显存从 28GB 降到 7GB；② 开启 KV Cache 分页，显存占用再降 20%；③ 用 vLLM 推理框架，显存占用再降 10%，速度提升 3 倍；最终 7B 模型在消费级显卡（RTX3090/4090）上就能部署，显存仅需 6~8GB。

#### 6.常用的大模型推理框架有哪些？选型依据？

答：① vLLM：目前最优解，速度快、显存低、支持长文本，应届生项目首选；② TensorRT-LLM：英伟达官方框架，速度最快，适合英伟达显卡；③ TGI：HuggingFace 官方框架，易用性高，适合快速落地；④ Ollama：轻量级，一键部署，适合本地测试；应届生项目用 Ollama+vLLM 即可。

7. #### 大模型部署的常用格式？ONNX/TensorRT/Safetensors 的区别？

答：① Safetensors：安全、快，适合模型权重存储，是目前主流；② ONNX：跨平台，适合推理部署，支持多框架；③ TensorRT：英伟达的推理引擎，速度最快，适合英伟达显卡；应届生项目里用 Safetensors 存储模型，用 TensorRT 做推理加速即可。

8. #### 什么是模型蒸馏？和量化的区别？

答：蒸馏是把大模型的知识迁移到小模型里，比如把 70B 模型的知识蒸馏到 7B 模型里，小模型的效果接近大模型，速度更快；量化是降精度，蒸馏是降参数量；两者可以结合使用，效果更好。

 

## 六、大模型八股拓展 + 高频杂题（12 道，必考，查漏补缺）

#### 1.大模型产生幻觉的原因？怎么解决？【★★★】

答：原因：① 预训练知识不完整 / 过时；② 生成时的自回归特性，模型会编造内容填补空缺；③ 训练数据有噪声；④ 指令理解偏差；解决（按优先级排序）：① RAG 检索增强（最有效）；② 微调对齐（SFT/DPO）；③ 优化 prompt，限定模型回答范围；④ 加入事实校验模块，对生成的内容做验证。

2. #### 大模型的长文本处理有哪些方案？【★★★】

答：① RoPE 插值：把模型的位置编码插值到更长的长度，比如把 4k 长度的模型适配到 8k；② PageAttention：把 KV Cache 分页，解决长文本显存爆炸；③ 滑动窗口注意力：只关注最近的窗口内的 token，降低计算量；④ 文档分块：把长文档分成多个短块，用 RAG 检索后拼接生成。

3. #### 大模型的参数量（7B/13B/70B）的区别？选型依据？

答：7B：显存低、速度快，适合本地部署和应届生项目，效果足够应对大部分场景；13B：效果比 7B 好，显存要求稍高；70B：效果最好，显存要求极高，适合企业级项目；应届生项目用 7B 即可，面试时说清楚选型理由：兼顾效果和效率。

4. #### 大模型的安全对齐有哪些方法？

答：① RLHF/DPO：对齐人类偏好，让模型输出安全内容；② 指令过滤：过滤敏感指令，比如暴力、色情；③ 输出审核：对生成的内容做审核，过滤敏感信息；④ 模型微调：用安全数据集微调，让模型拒绝生成敏感内容。

5. #### 多模态大模型（文本 + 图像）的核心原理？

答：核心是「图像 Encoder + 文本 Decoder」，比如 GPT-4o，用图像 Encoder 把图像转成向量，和文本向量拼接后，输入到大模型里生成答案；应届生不用深挖，面试时说清楚这个架构即可。

6. #### 为什么大模型的训练要用千亿级别的数据？

答：因为大模型是「涌现能力」，参数量越大、数据量越多，模型的理解和生成能力越强，当数据量达到一定阈值后，模型会出现质的飞跃，比如能做推理、总结、创作等复杂任务。

## 知识图谱相关：

#### 一、 知识图谱基础核心【★★★ 必考，概念类，全部背熟】

1.问：什么是知识图谱？核心组成是什么？

答：知识图谱（KG）是结构化的语义知识库，核心是以「实体 - 关系 - 实体」的三元组形式存储知识，把碎片化的文本知识转化为有逻辑、有关联、可推理的结构化图谱；核心组成有 3 个：① 实体：客观存在的事物（如高血压、阿司匹林、脑梗）；② 关系：实体之间的关联（如并发症、治疗、预防）；③ 属性：实体的特征（如高血压是慢性病、高发人群为中老年人）。

 

#### 2.问：知识图谱的核心存储形式是什么？为什么用这种形式？

答：核心存储形式是三元组 (头实体，关系，尾实体)，例如(高血压, 并发症, 脑梗)、(阿司匹林, 治疗, 高血压)；原因是：这种形式能清晰表达实体间的逻辑关联，支持精准的关联查询和推理，且存储结构简单，适配图数据库的检索逻辑，查询效率远高于非结构化文本。

 

#### 3.问：知识图谱的核心能力有哪些？最大的优势是什么？

答：核心能力：① 精准的关联推理能力，能推导实体间的隐含关系；② 零幻觉输出，所有知识均有结构化来源，不会编造内容；③ 可解释性强，回答问题时能给出明确的推理路径；④ 高效的结构化查询，适配 “谁 / 什么 / 哪里 / 为什么” 的精准问答场景。最大优势：零幻觉 + 可解释 + 精准推理，这正好弥补大模型和 RAG 的核心短板。

 

3. #### 问：知识图谱和传统的关系型数据库（MySQL）有什么区别？各自的适用场景？

   答：① 存储结构：MySQL 是「表结构」，存储二维行列数据，强调实体的属性；知识图谱是「图结构」，存储三元组，强调实体的关系；② 查询能力：MySQL 擅长结构化的条件查询（如查某类疾病的患者），不擅长多跳关联查询（如查高血压→并发症→脑梗的关联）；知识图谱擅长多跳推理和关联查询，精准适配知识类问答；③ 适用场景：MySQL 适合业务数据存储（如用户信息、订单），知识图谱适合语义化的知识存储与推理（如医疗 / 法律 / 金融的知识库问答）。

 

## 二、 知识图谱 & 大模型 & RAG 对比与结合【★★★ 最高频必考，面试 100% 问到，重中之重，金句级答案】

#### 1.问：知识图谱（KG）、大模型（LLM）、RAG 三者的核心区别是什么？各自的优缺点？【★★★ 春招原题，必背】

答：三者是互补关系，核心差异在于知识的存储形式和能力边界，也是目前大模型落地的三大核心组件：

大模型（LLM）：是无结构化的黑盒知识库。优点：理解自然语言、生成流畅文本、适配开放式创作；缺点：幻觉严重、知识不可控、推理能力弱、答案无解释性、知识过时。核心定位：擅长「理解和生成」，不擅长「精准推理和知识复用」。

RAG 检索增强生成：是非结构化的文本检索库。优点：低成本解决大模型的知识时效性和部分幻觉问题，无需微调即可适配垂直领域；缺点：检索结果是文本片段，无逻辑关联、无法推理、语义相似≠逻辑相关，检索准确率有限。核心定位：给大模型「补充碎片化文本知识」，解决 “不知道” 的问题。

知识图谱（KG）：是结构化的逻辑推理库。优点：零幻觉、精准推理、可解释、查询效率高；缺点：无法理解自然语言、无法生成流畅文本、构建成本高、适配不了开放式问题。核心定位：给大模型「补充结构化逻辑知识」，解决 “推理错” 的问题。

#### 2.问：大模型 + RAG + 知识图谱 三者怎么结合？为什么这是目前最优的落地方案？【★★★ 必考，必背，可直接结合你的医疗项目作答】

答：① 核心结合流程（我的医疗项目就是这么落地的）：用户提问 → 先通过知识图谱做结构化精准检索 + 推理，得到核心逻辑答案和推理路径 → 再通过 RAG 做非结构化文本检索，得到详细的文本补充信息 → 最后将「图谱的结构化结果 + RAG 的文本片段」拼接成 prompt，传给大模型做自然语言生成，输出流畅、准确、无幻觉、可解释的最终答案。② 为什么是最优解：三者能力互补，完美解决各自的短板 —— 大模型弥补了知识图谱的 “理解和生成” 短板，RAG 弥补了大模型的 “文本知识时效性” 短板，知识图谱弥补了大模型和 RAG 的 “推理和幻觉” 短板；这也是 2025 年大厂大模型落地的标配方案。

#### 3.问：纯 RAG 和「RAG + 知识图谱」的效果差异是什么？在你的项目中有什么体现？【★★★ 必考，结合项目加分】

答：① 纯 RAG 的痛点：检索结果是文本片段，无逻辑关联，只能做相似度匹配，回答复杂问题时准确率低，比如问 “阿司匹林为什么能预防脑梗”，纯 RAG 只能返回零散的文本，无法给出逻辑推理；② RAG + 知识图谱的优势：先通过图谱得到「阿司匹林→治疗高血压→预防脑梗」的逻辑路径，再通过 RAG 补充阿司匹林的剂量、禁忌症等文本细节，答案的准确率和完整性大幅提升；③ 项目体现：我的医疗 RAG 项目中，加入知识图谱后，问答准确率从 92% 提升至 98%，幻觉率从 5% 降至 0，复杂推理类问题的回答质量提升最明显。

#### 4.问：大模型能不能替代知识图谱？为什么？【★★★ 高频追问】

答：不能替代，二者是互补关系，不是替代关系。原因：① 大模型的核心能力是「理解和生成」，推理能力弱，且存在幻觉，无法给出可解释的答案；② 知识图谱的核心能力是「精准推理和结构化存储」，但无法理解自然语言，也无法生成流畅的文本；③ 大模型可以辅助知识图谱的构建（比如抽取实体和关系），知识图谱可以弥补大模型的知识短板，二者结合才能发挥最大价值。

## 三、 知识图谱的构建与落地【★★★ 必考，工程类，贴合应届生能力，无超纲内容】

#### 1.问：知识图谱的完整构建流程是什么？应届生能落地哪部分？【★★★ 必考】

答：完整构建流程分为 4 步，应届生只需掌握核心落地环节，无需掌握复杂的工业级优化，也是我项目中用到的流程：① 知识抽取：从非结构化文本（如医疗 PDF）中抽取「实体、关系、属性」，形成三元组；应届生落地方式：用大模型（如 Qwen/ChatGLM）做文本抽取，无需学复杂的 NLP 算法，效率高、效果好。② 知识融合：对抽取的知识做清洗、去重、对齐（比如统一实体名称），保证知识的一致性。③ 知识存储：将三元组存入图数据库 / 图工具，应届生首选：NetworkX（本地代码，无部署，适合项目开发）、Neo4j（轻量级图数据库，Docker 一键启动）。④ 知识应用：基于图谱做检索、推理、问答，对接大模型和 RAG，形成完整的问答系统。

#### 2.问：知识抽取的主流方式有哪些？各自的优缺点？你在项目中用的哪种？【★★★ 必考】

答：主流有 3 种方式，我在项目中用的是大模型抽取，也是应届生最优选择：① 规则抽取：基于正则表达式、关键词匹配抽取，优点：速度快、成本低；缺点：规则覆盖率低，适配不了复杂文本，泛化性差。② 传统 NLP 模型抽取：基于 BERT/ERNIE 训练抽取模型，优点：效果比规则好；缺点：需要标注数据，训练成本高，应届生落地难度大。③ 大模型抽取：用大模型做 prompt 指令抽取，优点：无需标注数据、适配所有垂直领域、抽取效果好，应届生易落地；缺点：推理速度略慢，但完全满足项目需求。补充（项目举例）：我在医疗知识图谱项目中，用 Qwen-7B 做抽取，输入 prompt「从以下文本中抽取实体和关系，格式为 (头实体，关系，尾实体)」，即可快速得到三元组，效率极高。

#### 3.问：知识图谱的主流存储工具 / 图数据库有哪些？各自的适用场景？你在项目中用的哪种？【★★★ 必考】

答：主流工具分两类，应届生只需掌握轻量级工具，也是面试高频考点，我在项目中做了分层选型：

轻量级本地工具：NetworkX（Python 库），无需部署任何服务，直接写代码构建图谱、做检索和推理，适合本地项目开发、应届生练手，我在项目中就是用这个工具构建医疗图谱。

入门级图数据库：Neo4j，开源、易用，支持图查询语言 Cypher，适合中小型项目的落地，Docker 一键启动，无需复杂配置。

工业级图数据库：NebulaGraph（国内大厂标配）、JanusGraph，分布式部署，支持亿级规模的图谱存储，适合企业级生产环境，应届生只需了解名称即可，无需掌握部署。

核心选型逻辑：小体量项目用 NetworkX，中体量项目用 Neo4j，大体量企业级项目用 NebulaGraph。

#### 4.问：你在项目中是怎么实现知识图谱的检索和推理的？【★★★ 必考，结合代码作答】

答：我在项目中基于 NetworkX 实现了基础的精准检索 + 多跳推理，适配医疗领域的问答场景，核心逻辑分为 2 类：① 精准检索：根据「头实体 + 关系」查询尾实体，比如输入「高血压 + 并发症」，返回脑梗、冠心病等结果；② 多跳推理：基于实体的关联关系做链式推理，比如从「阿司匹林→治疗→高血压」和「高血压→并发症→脑梗」，推理出「阿司匹林→间接预防→脑梗」；实现方式：用 Python 的 NetworkX 构建有向图，存储三元组的关联关系，通过遍历图的邻居节点实现检索和推理，代码简洁，无需复杂算法，效果精准。
